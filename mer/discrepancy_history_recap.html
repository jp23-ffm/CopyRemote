How Discrepancy History actually works

Every night's run of analyze_discrepancies reads the live server inventory, classifies each server, and writes three separate, never-mixed populations into a handful of tables. This page traces exactly how, so a defective run can be diagnosed — and safely deleted.

1 How one server gets classified

Three checks run against inventory.Server, each against its own population. Because the populations are carved up by LIVE_STATUS and SNOW_STATUS, a given server can only ever land in one of the three outcomes below — that's why missing-data, alive-inconsistent and dead-inconsistent are structurally incapable of overlapping.

The two dashed "gap" branches are real. A server with, say, LIVE_STATUS = ALIVE and SNOW_STATUS = "Planned" matches none of the three checks — it's neither flagged nor counted as clean anywhere. Worth knowing when a number looks lower than expected.

3 Where each number actually lives

Six tables, four roles. Same color-coding as the diagrams above.

TableRoleWritten byRead by
ServerDiscrepancy raw Flat dump of this run's problem servers, all field values inline. Truncated and rewritten every run — no history. bulk_insert_discrepancies() Live dashboard, Servers page, dashboard_filter_api
DiscrepancyTracking tracker One row per server with an active issue. active_issues JSON keeps a first_seen per field — the only place "how long has this been broken" lives. update_tracker() filter_persistent_records(), the days_open column/filter everywhere
AnalysisSnapshot aggregate One row per run: totals, the 3 persistent counts, diff vs previous run. create_analysis_snapshot() History page header, the 3 metric cards, dashboard trend chart
AnalysisSnapshotBreakdown aggregate Per-dimension counts (region, OS, live status, …) — missing_data metric only, one row per (dimension, value). save_breakdowns() The 8 "By …" panels under Missing data
AnalysisSnapshotCrossBreakdown aggregate row_field × OS-bucket matrix. row_field="REGION" for the Region×OS table (all 3 metrics) and row_field="RECAP" for the compact per-field recap (missing_data only, row_value = field name instead of a region). save_cross_breakdown() Region×OS tables, Missing data recap
ImportStatus Success/failure log line per run, plus the safety-abort message if the ±10% delta check trips. Command.handle() Logs / imports page

4 What breakdown_groups.json controls config

Four keys, each changeable without touching Python.

KeyControls
dimensionsWhich fields get a full "By …" panel (missing_data only). Add an entry → new panel, no code change.
groups.OSFAMILYHow raw OSFAMILY values bucket into Windows / Linux / Other columns everywhere OS-bucket columns appear. Case-insensitive.
metrics.<metric>.breakdown"dimensions" (full per-dimension panels) or "cross" (Region×OS only). Alive/dead are "cross" — no per-value detail.
recap.row_fields / recap.bucket_fieldWhich checked fields get a row in the Missing data recap. Must be a member of FIELDS_TO_CHECK or the row is always zero.

5 Deleting a defective snapshot

Safe, and simpler than it looks — you only ever delete one row.

from discrepancies.models import AnalysisSnapshot
AnalysisSnapshot.objects.filter(pk=<id>).delete()

AnalysisSnapshotBreakdown and AnalysisSnapshotCrossBreakdown both point at AnalysisSnapshot with on_delete=CASCADE — deleting the snapshot deletes every breakdown and cross-breakdown row that belongs to it. Nothing manual.

Leave DiscrepancyTracking alone. It isn't tied to a snapshot — it's the running memory of "how long has this issue been open" for whatever's currently broken. Deleting it would reset the day-count for every still-active issue, not just the bad run.

Finding the id: from the History page URL (?date=…&snapshot=<id> once you've picked a specific run from the dropdown), or:

AnalysisSnapshot.objects.order_by('-analysis_date').values('id', 'analysis_date', 'servers_with_issues')[:10]
Reflects the current schema — 3-way metric split, config-driven dimensions, per-metric cross-breakdowns, and the field-only recap table.
