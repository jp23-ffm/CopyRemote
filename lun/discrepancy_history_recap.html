How Discrepancy History works

A walkthrough of the daily pipeline that turns the live inventory into the History page: what runs, what gets written where, what gets thrown away every day, and what survives.

rewritten every run — no history
the memory — one row per server, updated in place
written once per day, kept forever
config file — no code change to extend
01

The big picture

Everything starts from one command, run once a day. It reads the live inventory and fans out into two very different kinds of tables — ones that get wiped every run, and ones that accumulate.

Source
Server inventory

inventory.Server — ~58k rows in dev, ~5k physical servers in prod. Nothing here is written by this pipeline; it's read-only input.

Trigger · daily
manage.py analyze_discrepancies

Runs 3 checks (missing fields, ALIVE-but-retired, DEAD-but-operational) against every eligible server and builds a list of "problem" records for today.

Written every run
rawServerDiscrepancy
Deleted and reinserted from scratch, every run. Today's state only — this is what the live dashboard queries.
memoryDiscrepancyTracking
One row per server. Updated, not replaced: a new issue gets first_seen = now; a fixed one is removed. This is the only place "how long has this been broken" is remembered.
Filter
Keep only issues open ≥ N days

Reads DiscrepancyTracking.oldest_first_seen to drop anything that's just today's noise. Default N = 7, set with --persistent-days. See the persistence filter below.

Written once per day · kept forever
dailyAnalysisSnapshot
One row for today. Global counts, raw and persistent.
dailyAnalysisSnapshotBreakdown
One row per (dimension, value) — region, OS, live status…
dailyAnalysisSnapshotCrossBreakdown
Region × OS-bucket matrix.
config which dimensions and buckets exist — breakdown_groups.json
Read by
History page & Excel export

Pick a date, land on the closest snapshot on or before it, render its rows. Nothing is recomputed on the fly.

02

The daily run, step by step

What analyze_discrepancies actually does, in order. Each step writes to (or reads from) one of the tables above.

#StepDoesTouches
1analyze_servers()Runs the 3 checks against inventory.Server, builds records — one entry per server with at least one issue.read Server
2compute_diff()Compares today's records against the current ServerDiscrepancy, before it gets wiped — produces new / resolved / changed lists.read ServerDiscrepancy
3bulk_insert_discrepancies()Deletes all of ServerDiscrepancy, reinserts today's records. No history kept here on purpose.write ServerDiscrepancy
4update_tracker()New issues get first_seen = now; still-open issues keep their original date; fixed issues are removed (row deleted if nothing's left).write DiscrepancyTracking
5filter_persistent_records()Keeps only records whose server has an issue open ≥ persistent_days in the tracker. This is the "real" discrepancy list.read DiscrepancyTracking
6create_analysis_snapshot()Writes one row for today: raw counts (for the live dashboard) and persistent_* counts (for History), side by side.write AnalysisSnapshot
7compute_breakdowns()For each dimension listed in breakdown_groups.json, groups the persistent records and counts population vs. issues per value.write AnalysisSnapshotBreakdown
8compute_cross_breakdown()Buckets each server's OS into Windows / Linux / Other (per config) and crosses it with region.write AnalysisSnapshotCrossBreakdown

Server A's issue never reaches the breakdown or snapshot tables at all. Server B's issue is invisible to History for its first 6 days, then included from day 7 onward — both AnalysisSnapshot and its breakdowns only ever see the day-7-plus state.

05

The config file drives the page

Every "By …" block on the History page comes from one JSON array, read by both the command (what to compute) and the view (what to show, and its label). Add an entry, get a new block — no code change.

{
  "dimensions": [
    { "field": "REGION", "label": "By region" },
    { "field": "OSSHORTNAME", "label": "By OS" },
    { "field": "LIVE_STATUS", … },
    … 5 more
  ],
  "groups": {
    "OSFAMILY": {
      "buckets": {
        "Windows": ["Windows"],
        "Linux": ["Linux"]
      },
      "other_label": "Other" // Solaris, AIX, …
    }
  }
}
1Panel “By region”
2Panel “By OS”
3Panel “By live status”
…5 more panels, same pattern
⊞Region × OS matrix columns: Windows · Linux · Other

Anything not explicitly listed in a bucket falls back to other_label automatically — adding VMware later means adding one line, not touching "Other".

Guardrail: the command logs a warning if a dimension turns out to have more than 50 distinct values — a sign it's not a good fit for this table.

06

Viewing a date

Nothing is computed on request — the History page and its export just read whichever day's rows already exist.

Date picker
?date=2026-07-10
→
_resolve_snapshot()
Latest AnalysisSnapshot on or before that date
→
Related rows
snapshot.breakdowns + .cross_breakdowns
→
Render
HTML panels, or Excel: Summary + 1 sheet / dimension + matrix

If the exact date has no snapshot (weekend, missed run…), it silently falls back to the closest earlier one and flags the mismatch on the page.

07

Why aggregates, not raw history

The tables above stay small on purpose — nothing here stores a per-server row for every day.

~60 rows/day
what's actually written — AnalysisSnapshot + its breakdowns
~5,000 rows/day
what a full per-server history would write, at prod scale
~1 GB/yr
estimated cost of the raw version on Postgres — affordable, but not needed for the trend the team actually wants to see
